# Parser edge case files

Files that exercise specific parser quirks, boundary conditions, or
backwards-compatibility behaviors. Each file is documented with its source,
what it tests, and the expected parser behavior.

## Files

### `23andme_lookalike_rejected_by_PA.txt`
- **Source:** openSNP user1, file 2995. Provenance: CC0.
- **What it is:** A 3-genotype file ("Tipsy Test" — alcohol metabolism panel
  testing rs1229984 ADH1B and rs671 ALDH2) shaped like 23andMe (4 tab columns)
  but with a single user-written comment `# results of my Tipsy Test (courtesy
  of BGI)` as the only header. Not actually from 23andMe.
- **What it tests:** The v1.1.0 P-A canonical-header tightening. Before P-A
  the 23andMe parser auto-detected on any "23andMe" substring anywhere in
  comments — this file's "# results of my Tipsy Test" wouldn't have triggered
  detection (no "23andMe" mention), but other lookalikes would have. P-A
  requires the canonical `# This data file generated by 23andMe` prefix
  before accepting the file.
- **Expected behavior:** `allelix stats` returns `Error: No parser recognized`.

### `23andme_format_from_genes_for_good_service.txt`
- **Source:** openSNP user4735, file 4081. The "Genes for Good" research
  service (University of Michigan) exports user data in 23andMe format.
  Provenance: CC0.
- **What it is:** A genuine 23andMe-format file (canonical first-line
  `# This data file generated by 23andMe at: ...`) exported by a different
  service. Despite the filename's `.genes-for-good.txt` suffix, the file
  contents are pure 23andMe format.
- **What it tests:** Parser format auto-detection is content-driven (not
  filename-driven). A file exported by a non-23andMe service in 23andMe
  format must still parse cleanly as 23andMe.
- **Expected behavior:** `allelix stats` reports format `23andMe`, build
  `GRCh37`, ~960K variants.

### `ftdna_grch36_positions.csv`
- **Source:** openSNP user1084, file 597 (FTDNA). Provenance: CC0.
- **What it is:** An FTDNA file with chromosomal positions on the GRCh36
  reference, not GRCh37. FTDNA files don't carry a build declaration in any
  header, so the parser has no in-band signal to distinguish builds.
- **What it tests:** A known limitation tracked as roadmap item R-12.
  FTDNA's `build_detect` probe table has GRCh37/38 entries but the v1.1.0
  P-B GRCh36 addition isn't wired into the FTDNA path (FTDNA has no header
  at all). Until R-12 lands, this file is silently labeled as GRCh37 even
  though its positions are GRCh36.
- **Expected behavior (current):** `allelix stats` reports `Build: GRCh37`
  (incorrect but documented). Annotations work for rsID-only sources
  (PharmGKB/GWAS/SNPedia) but ClinVar dispatches by build and produces
  zero rows.
- **Expected behavior (post-R-12):** `Build: GRCh36` detected from
  positions, with the GRCh36 unsupported-database warning fired.

### `unsupported_decodeme.txt`
- **Source:** openSNP user3400, file 2218. Provenance: CC0.
- **What it is:** A genotype file from deCODEme (Iceland-based genetic
  testing service, defunct as of 2013). Not in any supported format.
- **What it tests:** Unsupported format is detected and rejected cleanly.
- **Expected behavior:** `allelix stats` returns `Error: No parser recognized`.

### `unsupported_23andme_exome_vcf.txt`
- **Source:** Corpas family "Corpasome" release; dataset via figshare,
  project tag "Corpasome". Originally deposited on openSNP as user1,
  file 3302.
- **Citation:** Corpas M et al., "Crowdsourced direct-to-consumer genomic
  analysis of a family quartet," *BMC Genomics* 2015.
  doi:10.1186/s12864-015-1973-7
- **License:** Released CC-BY by the family (2013); described as CC0 in
  the 2015 paper. Attributed here to satisfy CC-BY.
- **What it is:** Whole-exome variant calls (~115k variants) from a
  GATK HaplotypeCaller pipeline. VCF format, not supported by Allelix
  v1.x (VCF is on the v2 roadmap).
- **What it tests:** Unsupported format rejected cleanly even when the
  filename suggests it might be 23andMe.
- **Expected behavior:** `allelix stats` returns `Error: No parser recognized`.
- **Note:** This is the only named-individual dataset in the repo. The
  Corpas family deliberately released their genomic data to the public.
  Replacement with a synthetic or fully anonymous exome VCF is tracked
  as a follow-up.

### `mhg_grch38_with_grch37_header.txt`
- **Source:** Synthetic MHG (Tempus) fixture. Hand-built from the project's
  `tests/fixtures/mock_myhappygenes_mislabeled.txt` mock-data generator
  output (committed to the repo; expanded here for the library).
- **What it is:** A 2,000-variant MHG-format file with positions on GRCh38
  but a `MyHappyGenes [TEMPUS]` header that claims `build 37.1`. Replicates
  the build-mismatch pattern that MHG/Tempus exports have been observed to
  ship with (header lies about build; positions are actually on the newer
  reference).
- **What it tests:** ADR-0021 — the position-based build auto-detector
  identifies GRCh38, the parser sees the header claim of GRCh37, and the
  `Build mismatch:` warning fires. The detected build (GRCh38) is used for
  annotation dispatch; the (wrong) header build is ignored.
- **Expected behavior:** `allelix analyze` prints:
  - `Build: GRCh38 (detected; 3/3 known-SNP positions matched)`
  - `Build mismatch: file header claims GRCh37 but position data is GRCh38. Using GRCh38.`

## Adding new edge cases

When a parser bug surfaces in production data, save the offending file here
with a descriptive name and a README entry. Include:

- Source (which openSNP user / file number if applicable)
- What about the file triggers the parser quirk
- Expected behavior (before and after the fix)
- Issue or PR number that addresses it (if applicable)

Edge case files exist to prevent regressions — they are the test corpus that
catches what fixtures miss.
